# 상세 화면·모바일 카테고리·세션·태그 개선 설계

## 목표

이미지 상세 화면의 불필요한 파일 정보를 정리하고 등록일을 표시한다. 네이버지도 링크의 호버 피드백을 분명하게 만들고, 모바일 카테고리 관리 행의 깨지는 배치를 바로잡는다. 웹 세션이 만료되면 현재 경로를 보존한 로그인 화면으로 이동하며, 이후 AI 분석부터 의미가 겹치지 않는 2~5개 태그를 생성하고 식당명은 태그에서 제외한다.

## 범위

- 이미지 상세 화면 정보 행과 네이버지도 메타데이터 링크 스타일
- 설정 화면의 모바일 카테고리 관리 행
- 웹 API 클라이언트의 Bearer 세션 만료 처리
- AI 태그 응답 계약·프롬프트와 식당명 태그 방어 처리
- 관련 자동 테스트, AI/UI 문서, `PROGRESS.md`

기존 북마크 태그의 일괄 정리, 카테고리 DND 동작 변경, 로그인 방식 변경, 데이터베이스 변경은 범위 밖이다.

## 이미지 상세 화면

상세 화면 하단의 파일명, 해상도, 원본 파일 크기, AI 분석 상태를 모두 제거한다. 같은 위치에는 북마크의 `createdAt`을 `ko-KR` 로컬 날짜 형식으로 변환한 `등록일`만 표시한다. 원본 보기와 다운로드 동작, 이미지 미리보기, 제목·설명·카테고리·메타데이터·태그는 유지한다.

날짜 형식은 브라우저의 한국 로캘을 사용하되 시간은 표시하지 않는다. 잘못된 날짜는 API의 공유 Zod 경계에서 이미 차단되므로 화면에 별도 폴백을 추가하지 않는다.

## 네이버지도 호버

범용 메타데이터 렌더러는 유지한다. URL key가 정확히 `네이버지도`일 때만 호버 배경을 네이버 녹색(`#03c75a`), 글자를 흰색으로 전환한다. 기본 상태와 다른 URL 메타데이터의 파란 스타일은 바꾸지 않는다. 키보드 포커스 링은 기존 전역 스타일을 그대로 사용한다.

## 모바일 카테고리 관리

생성 폼, 편집 저장 시점, 삭제 확인, DND 센서와 서버 mutation은 변경하지 않는다. 카테고리 행만 모바일에서 다음 2단 구조로 배치한다.

- 왼쪽: 두 행 높이를 차지하는 드래그 핸들
- 오른쪽 첫 행: 가용 폭 전체를 사용하는 이름 input
- 오른쪽 둘째 행: 북마크 개수와 우측 삭제 버튼

`sm` 이상에서는 현재의 한 줄 `핸들 / 이름 / 개수 / 삭제` 배치를 유지한다. 이를 위해 공용 `SortableRow`에 선택적 handle class를 전달할 수 있게 하되 기존 소비자의 기본 스타일은 바꾸지 않는다. input에는 `min-width: 0`을 보장해 375px 이하에서도 overflow가 생기지 않게 한다.

## 세션 만료 리다이렉트

인증 만료 판단은 웹 API 클라이언트에 집중한다. 다음 상황은 복구 불가능한 웹 세션 만료로 처리한다.

1. 요청 직전 `getSession()` 결과에 access token이 없음
2. 최초 API 401 뒤 `refreshSession()`이 실패하거나 새 access token을 반환하지 않음
3. 새 token으로 한 번 재시도한 응답도 401

이 경우 현재 `pathname + search`를 `redirect` 쿼리로 보존한 `/login` URL을 만들고 `window.location.assign()`으로 이동한다. 동시에 401 `ApiClientError`를 throw해 navigation이 즉시 완료되지 않는 테스트·브라우저 환경에서도 보호된 query가 정상 데이터처럼 계속되지 않게 한다.

API 5xx, 네트워크 실패, JWKS 조회 장애로 반환되는 502는 세션 만료로 취급하지 않는다. refresh는 최대 한 번만 수행한다. 기존 로그인 성공 후 안전한 내부 경로 복귀 규칙을 재사용하며 외부 URL은 허용하지 않는다.

## 태그 생성 정책

AI 응답의 태그 개수 계약을 3~5개에서 2~5개로 변경한다. 프롬프트에는 다음 규칙을 명시한다.

- 핵심 주제가 두 개면 두 개에서 멈추고 개수를 채우기 위한 태그를 만들지 않는다.
- 동의어·표기 변형은 하나만 선택한다. 예: `TrueNAS`와 `NAS`, `돈까스`와 `돈카츠`와 `미소카츠`를 동시에 만들지 않는다.
- 같은 의도의 상하위·조합형 태그와 여러 지역 맛집 변형을 나열하지 않는다. 예: `돈까스맛집`, `천호맛집`, `강동구맛집` 중 핵심 검색어만 남긴다.
- 식당 상호와 지점명은 태그에 넣지 않는다. 음식 종류·지역·방문 목적처럼 재사용 가능한 검색어만 사용한다.

AI의 의미 판단을 주 경계로 사용하되, 서버는 `place.name`과 공백·기호·대소문자를 무시했을 때 동일한 태그를 저장 직전에 제거한다. 이 방어 로직은 식당명만 다루며 일반 태그의 의미 중복을 문자열 휴리스틱으로 임의 삭제하지 않는다. 필터 후 태그가 두 개 미만이어도 식당명을 되살리지 않으며, 이후 분석 요청의 강화된 프롬프트가 정상적으로 2개 이상을 생성하게 한다. 기존 저장 데이터는 수정하지 않는다.

## 오류 처리와 접근성

- 상세 날짜와 메타데이터 링크는 기존 읽기 순서를 유지한다.
- 모바일 카테고리 행의 드래그 핸들·삭제 버튼은 기존 accessible name과 최소 터치 영역을 유지한다.
- 세션 리다이렉트는 로그인 화면에서 기존 `redirect` 파서로 검증한다.
- AI 태그 후보가 필터링되더라도 카테고리·제목·요약·메타데이터 분석 결과는 성공 처리한다.

## 테스트

- 이미지 상세 테스트: 등록일 표시, 파일명·해상도·크기·분석 완료 미표시
- 메타데이터 테스트: 네이버지도 전용 green hover class와 다른 URL 스타일 보존
- 설정 테스트: 모바일 2단 grid class와 데스크톱 한 줄 breakpoint class
- API 클라이언트 테스트: token 없음, refresh 실패/빈 세션, 재시도 401에서 내부 로그인 URL 이동과 401 error; 502는 이동하지 않음
- AI 테스트: 태그 두 개 응답 허용, 한 개 거부, strict JSON Schema `minItems: 2`, 중복 억제·식당명 제외 프롬프트
- 분류 테스트: 정규화된 `place.name`과 같은 태그만 제거하고 다른 태그와 분석 결과를 보존
- 전체 `bun run typecheck`, `bun run lint`, `bun run test`, `bun run build`

## 완료 기준

요청된 네 영역이 자동 테스트로 고정되고 전체 검증 루프가 통과해야 한다. 변경 이유와 검증 결과를 `PROGRESS.md`에 기록한다. 데이터베이스 migration이나 기존 태그 일괄 수정은 발생하지 않는다.
