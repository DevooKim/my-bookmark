# 05. AI 카테고리 분류

`packages/ai` — provider 중립 인터페이스. 기본 Gemini, Anthropic/OpenAI 전환 가능. 활성 provider와 provider별 API 키는 사용자 설정 DB가 원본이다.

## 인터페이스 (packages/ai/src/types.ts)

```ts
export interface CategorizeInput {
  url: string;
  title?: string;
  description?: string;
  siteName?: string;
  existingCategories: { id: string; name: string }[];
}

export type CategorizeResult =
  | { type: 'existing'; categoryId: string; confidence: number }
  | { type: 'new'; name: string; confidence: number }   // 새 카테고리 제안
  | { type: 'none' };                                    // 판단 불가 → 미분류 유지

export interface AiProvider {
  readonly name: string;
  categorize(input: CategorizeInput): Promise<CategorizeResult>;
}

export function createAiProvider(config: {
  provider: 'gemini' | 'anthropic' | 'openai';
  apiKey: string;
  model?: string;
}): AiProvider;
```

## Provider 구현

| provider | SDK | 기본 모델 |
|---|---|---|
| gemini (기본) | `@google/genai` | `gemini-flash-lite-latest` (2026-07: `gemini-2.5-flash` 서비스 종료로 rolling alias 사용) |
| anthropic | `@anthropic-ai/sdk` | `claude-haiku-4-5` |
| openai | `openai` | `gpt-4o-mini` |

구현 규칙:

- 각 SDK의 **구조화 출력 기능**을 사용한다 (Gemini: `responseSchema` + JSON mime, Anthropic: tool use 강제 호출, OpenAI: structured outputs). 응답은 반드시 zod 스키마로 파싱 — 실패 시 `{ type: 'none' }`으로 처리하고 warn 로그.
- 타임아웃 15초 (AbortSignal). 재시도는 하지 않는다 — 실패하면 `ai_status='failed'`로 남기고 사용자가 UI에서 재시도.
- **모델명은 시간이 지나면 바뀐다.** 구현 시점에 각 SDK 문서에서 현행 소형(저비용) 모델을 확인하고, 위 표와 다르면 코드 기본값을 현행으로 쓰되 PROGRESS.md에 기록.
- provider는 사용자별로 최초 분류 시 생성해 캐시하고, 설정 저장·키 삭제 시 해당 사용자 캐시를 무효화한다.

## 프롬프트 설계 (전 provider 공통 템플릿)

시스템 지시 요지:

```
너는 북마크 분류기다. 웹페이지 정보와 사용자의 기존 카테고리 목록을 보고
가장 적합한 카테고리를 고른다.

규칙:
1. 기존 카테고리 중 명확히 맞는 것이 있으면 반드시 그것을 선택한다 (id로).
2. 기존 카테고리로 무리 없이 분류할 수 없을 때만 새 카테고리를 제안한다.
   - 새 이름: 한국어, 1~10자, 일반적·재사용 가능한 수준 (예: "개발", "디자인", "요리")
   - 과도하게 구체적인 이름 금지 (예: "React 19 마이그레이션 글" ✗)
3. 정보가 부족해 판단이 어려우면 none을 반환한다.
4. confidence: 0~1.
```

유저 메시지에 url, title, description, siteName, 기존 카테고리 목록(id+name)을 넣는다. 페이지는 한국어/영어 혼재 가능 — 카테고리 이름 언어는 **기존 카테고리의 언어를 따르고**, 없으면 한국어.

## 분류 파이프라인 (apps/api/src/services/categorize.ts)

`POST /api/bookmarks (mode=ai)` 또는 `POST /api/bookmarks/:id/categorize` 후 백그라운드로 실행:

```
1. 메타데이터 fetch (아래) → bookmarks에 title/description/site_name/favicon/og_image 저장
   (mode와 무관하게 메타데이터는 항상 수집)
2. mode=ai일 때만:
   a. 사용자 카테고리 목록 로드
   b. provider.categorize(input)
   c. 결과 적용:
      - existing → category_id 세팅 (단, 반환된 id가 실제 존재하는지 검증)
      - new → 동일 이름 카테고리 존재 여부 재확인(대소문자 무시, trim) 후 생성 → 세팅
      - none → category_id 그대로(null)
   d. ai_status = 'done'
3. 어떤 단계든 예외 → ai_status = 'failed', 에러 로그. 북마크 자체는 유지
```

- fire-and-forget이지만 **반드시 `.catch()`로 종결**한다 (unhandled rejection으로 프로세스가 죽으면 안 됨).
- 적용 직전에 북마크가 여전히 `pending`인지 확인 (그 사이 사용자가 수동 분류했으면 결과 폐기 — `ai_status='idle'`이 됐을 것이므로 조건부 UPDATE `where ai_status='pending'`으로 자연스럽게 해결).

## 메타데이터 fetcher (apps/api/src/services/metadata.ts)

- Node 내장 fetch. 타임아웃 5초, redirect 최대 3회, 응답 본문 **1MB 제한**(스트림 읽다가 초과 시 중단).
- `User-Agent: MyBookmarkBot/1.0` 설정 (일부 사이트가 UA 없으면 차단).
- content-type이 `text/html`일 때만 파싱. 파서는 정규식 조합이 아닌 `cheerio` 같은 경량 파서 사용.
- 추출: `og:title` → `<title>` 순, `og:description` → `meta[name=description]`, `og:site_name`, `og:image`(절대 URL로 변환), favicon(`link[rel~=icon]` 절대 URL, 없으면 `https://www.google.com/s2/favicons?domain=<host>&sz=64`).
- 실패(타임아웃, 비HTML, 4xx/5xx)해도 치명적이지 않다 — 메타데이터 없이 진행하고 title은 URL의 host로 폴백.
- **SSRF 방지**: fetch 전에 호스트를 검사해 사설/루프백 IP(10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x, localhost, ::1)로의 요청을 거부한다. 1인 서비스라도 API Key 유출 시 내부망 스캔에 악용될 수 있는 경로다.

## 설정 요약

설정 화면에서 Gemini/Anthropic/OpenAI를 선택하고 provider별 API 키를 입력한다. 키는 `ai_settings`에 AES-256-GCM으로 암호화 저장하며, 암호화 마스터 키만 서버 환경변수로 둔다.

```
AI_SETTINGS_ENCRYPTION_KEY=…  # openssl rand -base64 32, 서버 전용
```

선택 provider의 API 키가 없으면 AI 비활성 모드로 동작한다(mode=ai 요청은 metadata만 수집하고 `ai_status='failed'`). provider나 키를 변경하면 서버 재시작 없이 다음 분류부터 적용된다. 모델은 provider별 코드 기본값을 사용한다.

## 테스트 (Vitest)

- 프롬프트/파싱 로직: provider별 mock 응답으로 3가지 결과 타입 + 파싱 실패 케이스.
- 파이프라인: provider를 fake로 주입해 existing/new/none/예외 각각에서 DB 상태 전이 검증 (supabase client는 얇게 mock).
- 메타데이터: 로컬 HTML fixture로 추출 검증 + SSRF 차단 검증. 실제 네트워크 호출 테스트는 만들지 않는다.
