# 05. AI 카테고리 분류

`packages/ai` — OpenRouter의 preset `@preset/my-bookmark` 단일 호출로 분류한다. 모델 선택·폴백 순서·요청 파라미터는 openrouter.ai 대시보드에서 preset으로 구성하며, 앱 코드는 어떤 모델이 응답했는지만 알면 된다. 키는 서버 env `OPEN_ROUTER_API_KEY` 하나뿐이다 — provider별 키 관리, 모델 카탈로그, 우선순위 DND, 폴백 체인 구현은 모두 이 아키텍처에 없다(과거 자체 provider 전환 방식에서 2026-07-12 OpenRouter preset 전환으로 대체).

## 인터페이스 (packages/ai/src/types.ts)

```ts
export interface CategorizeInput {
  url: string;
  title?: string;
  description?: string;
  siteName?: string;
  existingCategories: { id: string; name: string }[];
}

export type CategorizeResult =
  | { type: 'existing'; categoryId: string; confidence: number }
  | { type: 'new'; name: string; confidence: number }   // 새 카테고리 제안 (이모지+텍스트, 아래 참조)
  | { type: 'none' };                                    // 판단 불가 → 미분류 유지

export interface AnalyzeResult {
  category: CategorizeResult;
  summaryTitle: string;      // 한국어 제목 스타일, 최대 40자
  summary?: string | null;   // 한국어 1~3문장, 최대 300자. provider가 생략 가능
  tags: string[];             // 한국어 3~5개, 고유명사는 원문 허용
}

export interface AiProviderConfig {
  apiKey: string;
}

export interface AnalyzeOutcome {
  analysis: AnalyzeResult;
  model: string; // OpenRouter 응답의 실사용 모델 id (예: "google/gemini-3.1-flash-lite-20260507")
}

export interface AiProvider {
  categorize(input: CategorizeInput): Promise<AnalyzeOutcome>;
  validateConnection(): Promise<void>;
}

export function createAiProvider(config: AiProviderConfig): AiProvider;
```

`packages/ai`는 SDK를 쓰지 않는다 — `fetch` + zod로 직접 호출한다. OpenAI/Anthropic/Gemini SDK 의존성은 제거됐다(`provider.require_parameters` 같은 OpenRouter 확장 파라미터가 각 SDK 타입에 없어 `any` 없이는 표현이 불가했기 때문).

## Provider 구현 (packages/ai/src/providers.ts)

`PRESET_MODEL = "@preset/my-bookmark"` 하나만 존재한다. `OpenRouterProvider`는:

- `POST https://openrouter.ai/api/v1/chat/completions` — `model: PRESET_MODEL`, `max_tokens: 2048`(명시 필수 — 아래 실측 참조), `provider: { require_parameters: true }`(구조화 출력을 지원하는 업스트림으로만 라우팅하는 OpenRouter 확장 파라미터), `X-Title: my-bookmark` 헤더, strict `json_schema` 응답 포맷.
- 응답은 `openRouterCompletionSchema`(zod)로 파싱 후 `choices[0].message.content`를 JSON.parse → `parseAnalyzeResponse`(zod)로 최종 검증. 실패하면 throw(에러에 `status` 첨부 가능).
- `validateConnection()`은 `GET /key` 200 확인(추론 없음, 연결 테스트용).
- 타임아웃 15초 (Gemini/Anthropic/OpenAI 시절과 동일하게 `AbortSignal`, `validateConnection`은 10초). 재시도는 하지 않는다 — 실패하면 `ai_status='failed'`로 남기고 사용자가 UI에서 재시도(폴백 모델 전환은 openrouter.ai preset 설정이 담당 — 아래 참조).

## 실측으로 확인된 사실 (2026-07-12, 실제 키로 검증 완료)

1. `@preset/my-bookmark` 호출 성공 시 응답 `model` 필드에 실사용 모델(예: `google/gemini-3.1-flash-lite-20260507`)이 온다. 기존 카테고리 id 매칭, 이모지 신규 이름, 1~3문장 summary, 태그가 정확한 JSON으로 반환됨을 확인.
2. **strict `json_schema` 규칙 (400/402 재현 후 해결):**
   - OpenAI 계열 업스트림은 strict 모드에서 **모든 property가 `required`에 포함**돼야 한다. 조건부 필드(`categoryId`/`name`/`confidence`)는 `"type": ["string", "null"]`식 nullable로 선언하고 전부 required에 넣는다. 이 규칙을 임의로 되돌리지 말 것.
   - `max_tokens` 미지정 시 기본값이 매우 커서(65536) 일부 업스트림에서 402가 난다. **반드시 2048로 명시**한다.
3. `GET /api/v1/key`(inference 키로 호출 가능)는 `usage`, `usage_daily`, `usage_weekly`, `usage_monthly`(USD), `limit`, `limit_remaining`, `is_free_tier`를 반환한다 — 계정 사용액 카드(`GET /api/ai/account`)의 데이터 소스.
4. `/api/v1/activity`·`/api/v1/analytics/query`는 **management key 전용(403)** — 사용하지 않는다. 모델별·일별 비용 집계는 API로 불가하므로 상세 비용 분석은 openrouter.ai activity 페이지로 안내한다.

## 프롬프트 설계 (packages/ai/src/schema.ts systemPrompt())

시스템 지시 요지:

```
너는 북마크 분류기다. 웹페이지 정보와 사용자의 기존 카테고리 목록을 보고
가장 적합한 카테고리를 고른다.

규칙:
1. 기존 카테고리 중 명확히 맞는 것이 있으면 반드시 그것을 선택한다 (id로).
2. 기존 카테고리로 무리 없이 분류할 수 없을 때만 새 카테고리를 제안한다.
   - 새 이름은 "이모지 1개 + 공백 + 한국어 이름(1~10자)" 형식의 한 문자열
     (예: "💻 개발", "📰 뉴스", "🎨 디자인"). 이모지는 주제를 대표하는 것 1개만
     앞에 붙이고, 이름은 일반적·재사용 가능한 수준으로 한다.
3. 정보가 부족해 판단이 어려우면 category에 none을 반환한다.
4. confidence는 0~1이다.
5. summaryTitle은 원문을 핵심만 요약한 한국어 제목 스타일로 작성하며 최대 40자다.
6. tags는 중복 없는 한국어 태그 3~5개로 작성하며 각 태그는 최대 20자다. 고유 기술명은 원문 허용.
7. summary는 원문의 핵심을 한국어 1~3문장으로 요약한다. 불필요한 수식 없이
   정보만 담고, 전체 300자 이내로 한다.
8. 해당 없는 필드(categoryId, name)는 null로 채운다.
```

유저 메시지에 url, title, description, siteName, 기존 카테고리 목록(id+name)을 JSON으로 넣는다. 페이지는 한국어/영어 혼재 가능 — 카테고리 이름 언어는 **기존 카테고리의 언어를 따르고**, 없으면 한국어.

새 카테고리 이름의 중복 판정(`categorize.ts`의 `normalizeCategoryName`)은 선두의 이모지·기호를 제거하고 비교한다 — AI가 `"💻 개발"`을 제안했는데 기존에 `"개발"`이 있으면(또는 그 반대) 재사용한다. zod에서 신규 카테고리 이름은 `max(16)`(이모지 UTF-16 code unit + 공백 + 한국어 1~10자를 수용).

## 응답 zod 스키마 (packages/ai/src/schema.ts)

```ts
export const analyzeResponseSchema = z.object({
  category: categorizeResponseSchema, // existing/new/none discriminated union
  summaryTitle: z.string().trim().min(1).max(40),
  summary: z.string().trim().min(1).max(300).nullish(),
  tags: z.array(z.string().trim().min(1).max(20)).min(3).max(5)
    .refine((tags) => new Set(tags).size === tags.length),
});
```

`summary`는 optional/nullable로 파싱한다 — provider가 필드를 생략해도 분류 전체를 실패시키지 않고 `description` 갱신만 건너뛴다(과거 Gemini가 `confidence`를 생략했던 전례에 대한 동일한 방어). `confidence`도 null 허용 후 0으로 기본 처리한다.

OpenRouter로 보내는 `json_schema`(strict)는 이 zod 스키마와 별개로 관리한다 — strict 모드 규칙(전부 required + nullable) 때문에 zod의 optional 필드도 JSON Schema에서는 required로 선언하고 `null`을 허용한다. 자세한 형태는 `packages/ai/src/schema.ts`의 `jsonSchema` 상수를 참조(문서에 복제하지 않음 — 코드가 원본).

## 분류 파이프라인 (apps/api/src/services/categorize.ts)

`POST /api/bookmarks (mode=ai)` 또는 `POST /api/bookmarks/:id/categorize` 후 백그라운드로 실행:

```
1. 메타데이터 fetch (아래) → bookmarks에 title/description/site_name/favicon/og_image 저장
   (mode와 무관하게 메타데이터는 항상 수집)
2. mode=ai일 때만:
   a. 사용자 카테고리 목록 로드
   b. provider.categorize(input) — 1회 호출 (재시도/폴백 없음, openrouter.ai preset이 대신 처리)
   c. 성공 시 ai_usage_events에 { model: 실사용모델, status: 'success' } 기록,
      실패 시 { model: '@preset/my-bookmark', status: 'failed', errorCode } 기록
   d. 결과 적용:
      - existing → category_id 세팅 (단, 반환된 id가 실제 존재하는지 검증)
      - new → 이모지 무시 정규화 비교로 동일 카테고리 존재 여부 재확인 후 생성 → 세팅
      - none → category_id 그대로(null)
   e. 같은 pending 조건부 UPDATE에서 summaryTitle을 title에, summary를 description에
      (있으면), tags를 태그 배열에, 실사용 모델을 ai_model에 저장하고 ai_status='done'으로 전이
3. 어떤 단계든 예외 → ai_status = 'failed', 에러 로그. 북마크 자체와 기존 제목·태그·description 값은 보존
```

- fire-and-forget이지만 **반드시 `.catch()`로 종결**한다 (unhandled rejection으로 프로세스가 죽으면 안 됨).
- 적용 직전에 북마크가 여전히 `pending`인지 확인 (그 사이 사용자가 수동 분류했으면 결과 폐기 — `ai_status='idle'`이 됐을 것이므로 조건부 UPDATE `where ai_status='pending'`으로 자연스럽게 해결).
- **AI 재분류는 모든 북마크에서 상시 노출**된다(`pending`만 제외) — 과거 "failed일 때만" 제한을 없앴다. 재분류는 제목·요약·태그·카테고리를 덮어쓰므로 실행 전 웹 UI가 `window.confirm`으로 확인한다.
- 사용량 기록(`ai_usage_events` insert) 실패는 분류 결과에 영향을 주지 않는다 — `console.warn` 후 계속 진행.

## 메타데이터 fetcher (apps/api/src/services/metadata.ts)

- Node 내장 fetch. 타임아웃 5초, redirect 최대 3회, 응답 본문 **1MB 제한**(스트림 읽다가 초과 시 중단).
- `User-Agent: MyBookmarkBot/1.0` 설정 (일부 사이트가 UA 없으면 차단).
- content-type이 `text/html`일 때만 파싱. 파서는 정규식 조합이 아닌 `cheerio` 같은 경량 파서 사용.
- 추출: `og:title` → `<title>` 순, `og:description` → `meta[name=description]`, `og:site_name`, `og:image`(절대 URL로 변환), favicon(`link[rel~=icon]` 절대 URL, 없으면 `https://www.google.com/s2/favicons?domain=<host>&sz=64`).
- 실패(타임아웃, 비HTML, 4xx/5xx)해도 치명적이지 않다 — 메타데이터 없이 진행하고 title은 URL의 host로 폴백.
- **SSRF 방지**: fetch 전에 호스트를 검사해 사설/루프백 IP(10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x, localhost, ::1)로의 요청을 거부한다. DNS resolve 결과도 검사한다(hostname 문자열만으로는 내부 IP로 resolve되는 도메인을 막을 수 없음).

## 설정 요약 (설정 화면 · 대시보드)

키·모델 관리 UI는 없다 — 서버 env `OPEN_ROUTER_API_KEY`가 유일한 설정이다. 설정 화면의 AI 섹션은 상태 카드 하나로 축소돼 있다: `{ enabled, preset }`을 표시하고 "연결 테스트"(`POST /api/ai/test`) 버튼과 사용량 대시보드 링크만 제공한다.

```
OPEN_ROUTER_API_KEY=…  # https://openrouter.ai/keys 에서 발급, 서버 전용
```

키가 없으면 AI 비활성 모드로 동작한다(`mode=ai` 요청은 metadata만 수집하고 `ai_status='failed'`). 모델 선택·폴백 우선순위·요청 파라미터를 바꾸려면 openrouter.ai 대시보드에서 preset `@preset/my-bookmark`을 수정한다 — 서버 재시작이나 앱 배포가 필요 없다.

**AI 사용량 대시보드**(`/ai-usage`, 설정에서 이동)는 두 데이터 소스를 조합한다:
- 로컬 `ai_usage_events` 집계(모델별 성공/실패 횟수, 일별 호출 수, 최근 이벤트 목록) — "북마크와 연결된 호출/실패 이력"이 이 데이터의 고유 가치.
- OpenRouter 계정 사용액 카드(`GET /api/ai/account`) — 오늘/이번 주/이번 달 USD, free tier 여부, 잔여 한도.

**토큰·비용은 로컬에 저장하지 않는다.** 공식 수치는 OpenRouter 쪽이고(BYOK·캐시 할인 등으로 로컬 사본은 오차가 생길 수 있음), 모델별·일별 비용 상세 분석은 대시보드에서 openrouter.ai activity 페이지로 안내하는 외부 링크(`https://openrouter.ai/activity`)로 대체한다.

> **향후 확장(현재 범위 아님)**: management key(`OPEN_ROUTER_MANAGEMENT_KEY`)가 제공되면 `POST /api/v1/analytics/query`(metrics: total_usage/tokens_total, dimensions: model + date__day) 프록시 라우트를 추가해 모델별·일별 토큰/비용 집계 카드를 대시보드에 붙일 수 있다. 현재 구조(로컬 이벤트 집계 + `/key` 계정 카드)는 그대로 두고 카드만 추가되는 형태로 설계돼 있어 충돌 없음.

## 테스트 (Vitest)

- 프롬프트/파싱 로직: fetch mock 기반 응답으로 existing/new/none 3가지 결과 타입 + nullable 필드 파싱 + 파싱 실패/비2xx 케이스.
- 파이프라인: provider를 fake로 주입해 existing/new/none/예외 각각에서 DB 상태 전이 검증, 성공/실패 각각의 `ai_usage_events` 기록 내용 검증(supabase client는 얇게 mock).
- 메타데이터: 로컬 HTML fixture로 추출 검증 + SSRF 차단 검증. 실제 네트워크 호출 테스트는 만들지 않는다(단, 실키 스모크는 배포 전 수동 1회 — PROGRESS 참조).
